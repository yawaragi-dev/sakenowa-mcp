# v0.1.0 specification — Six read-only tools over a Sakenowa-mirrored Postgres

> **Schema naming aligned to the canonical Sakenowa-API mirror.** The tool SQL
> and Zod inputs use the table/column names a Sakenowa-mirrored Postgres
> actually has (`areas`/`area_id`, `brands`/`brand_id`, `flavor_charts`/`f1..f6`,
> camelCase inputs like `brandId`/`areaId`/`topK`). The authoritative,
> divergence-by-divergence reference is
> [`docs/specs/schema-audit-v0.1.1.md`](./schema-audit-v0.1.1.md); where the
> per-tool prose below still shows the original `sake_id`/`name_ja`/romaji-axis
> names, the audit + the code supersede it.

This document is the authoritative scope for the v0.1.0 release of `@yawaragi/sakenowa-mcp`. The issue tree on GitHub breaks this scope into tracer-bullet slices that each ship a working subset.

## Goal

Publish v0.1.0 to npm as `@yawaragi/sakenowa-mcp` with six well-defined MCP tools that turn Sakenowa queries into structured tool calls usable by Claude Desktop, the AI SDK's MCP client, and any other MCP consumer.

## Out of scope for v0.1.0

- Cross-beverage mappings (whisky / wine / beer descriptors → 6-axis flavor positions). These are hand-curated heuristics that belong in consuming apps, not in an OSS Sakenowa wrapper. See `AGENTS.md` "What this project is NOT".
- Ingest pipeline. The server reads from a Postgres that the consumer has populated. v0.1.0 does not ship an ingest CLI.
- Caching, rate-limiting, or memoisation. The server is stateless.
- Authentication. Anyone with `DATABASE_URL` can query.
- LLM-based reasoning of any kind.
- A `get_brewery_details` tool. Consumers can build the equivalent from `get_sake_details` for v0.1.0; standalone brewery lookup is a v0.2 concern.
- A `list_flavor_tags` tool. The 117-tag vocabulary is too long for a single tool result; consumers should let users disambiguate tags interactively. A `search_flavor_tags(query)` variant may land in v0.2.

## Transport

The server speaks MCP over one of two transports, chosen at startup by `MCP_TRANSPORT`:

- **stdio** (default) — one server per child process, for Claude Desktop / IDE consumers. Stdout is reserved for MCP framing; all logs and errors go to stderr.
- **http** — a long-running [Streamable HTTP](https://modelcontextprotocol.io/specification/2025-03-26/basic/transports) server, for consumers that can't keep a child process alive (e.g. a serverless web app). Stateless: a fresh MCP server + transport per request, all sharing one Postgres pool, so there are no sessions and no per-connection state. Returns plain `application/json` JSON-RPC responses (the tools are short synchronous reads — no SSE streaming). The transport and its `@hono/node-server` bridge ship transitively with `@modelcontextprotocol/sdk`; no extra dependency.

An unknown `MCP_TRANSPORT` value fails loud on stderr with a non-zero exit. Tool descriptions are English, in the discovery/learning register (no marketing language).

**No built-in auth or TLS on the HTTP endpoint.** The server binds plain HTTP and stays stateless / no-auth / no-business-logic. Auth and TLS are the consumer's responsibility at deploy time (Cloudflare Tunnel, a reverse proxy with a bearer token, a platform's TLS gateway); the server cannot enforce them.

## Configuration

- `DATABASE_URL` (required) — Postgres connection string (read-only role recommended). The schema is documented in "Expected DB schema" below. If unset at startup, the server logs an explanatory error to stderr and exits non-zero.
- `MCP_LOG_LEVEL` (optional, default `error`) — `silent` | `error` | `info` | `debug`. Anything other than `silent` writes to stderr only.
- `MCP_TRANSPORT` (optional, default `stdio`) — `stdio` | `http`.

HTTP mode only (ignored under stdio):

- `MCP_HTTP_PORT` (default `3030`), `MCP_HTTP_HOST` (default `0.0.0.0`), `MCP_HTTP_PATH` (default `/mcp`).

## Tool surface (exactly six)

Each tool is described below with its purpose, input shape, output shape, and the SQL intent. Zod schemas, error handling, and exact column lists are part of the implementation; the shapes here describe the contract.

### 1. `list_prefectures`

**Purpose:** Enumerate Japan's 47 prefectures so the LLM can suggest valid filter targets to the user.

**Input:** none.

**Output:** `Prefecture[]` where each Prefecture is `{ id: number, name_ja: string, name_romaji: string }`.

**SQL intent:** `SELECT id, name_ja, name_romaji FROM prefectures WHERE id <> 0 ORDER BY id`.

The `areaId: 0` "Other" sentinel is excluded.

### 2. `search_sakes_by_name`

**Purpose:** Resolve a free-text name (romaji or kanji) into one or more Sake records, for downstream "tell me about X" / "find a sake like X" flows.

**Input:** `{ query: string, limit?: number /* default 10, max 50 */ }`.

**Output:** `Sake[]` ordered by match quality, each `{ id: number, name_ja: string, name_romaji: string, brewery: { id, name_ja, name_romaji }, prefecture: { id, name_ja, name_romaji } }`.

**SQL intent:** case-insensitive `ILIKE` match across `name_ja` and `name_romaji`, joined to `breweries` and `prefectures`. Match-quality ordering can be exact-prefix-first then substring; refine in implementation.

### 3. `get_sake_details`

**Purpose:** Return everything the server knows about one Sake, identified by its Sakenowa `brand_id`.

**Input:** `{ sake_id: number }`.

**Output:** a single object combining the Sake record, its FlavorProfile (the 6-axis vector with axis labels), its FlavorTags, its Brewery, and the Brewery's Prefecture. If no row exists for the id, the tool returns an explicit "not found" result (not an error).

**SQL intent:** `sakes` joined to `flavor_profiles`, left-joined to `sake_flavor_tags` × `flavor_tags`, joined to `breweries` × `prefectures`.

### 4. `find_similar_sakes`

**Purpose:** "Sake similar to this one" — cosine similarity over the 6-axis FlavorProfile.

**Input:** `{ sake_id: number, top_k?: number /* default 10, max 50 */ }`.

**Output:** `Sake[]` ordered by descending similarity, each entry containing the same fields as a `search_sakes_by_name` result plus a `similarity: number` in `[0, 1]`.

**SQL intent:** read the source Sake's FlavorProfile vector; compute cosine similarity against all other Sake vectors with a flavor profile; ORDER BY similarity DESC LIMIT top_k. Implementation options:

- Plain SQL with explicit `(a1*b1 + a2*b2 + …) / (||a|| * ||b||)`. Works on any Postgres, no extensions required.
- `pgvector` extension if available, but **not required**. The mirror schema only stores six `numeric`/`float` columns; computing cosine in SQL is fast enough at the 5k-Sake scale.

Pick plain SQL for v0.1.0; pgvector is a v0.2 optimisation if benchmarks show it matters.

The source Sake itself is excluded from the result.

### 5. `find_sakes_by_flavor`

**Purpose:** Filter Sakes by flavor-axis position, flavor-tag membership, or prefecture, with combined `AND` semantics across the three filter families.

**Input:**

```
{
  axes?: {
    hanayaka?: { min?: number, max?: number },
    hojun?:    { min?: number, max?: number },
    juko?:     { min?: number, max?: number },
    odayaka?:  { min?: number, max?: number },
    dry?:      { min?: number, max?: number },
    keikai?:   { min?: number, max?: number }
  },
  tags?: number[],         // FlavorTag ids; ALL must be present on the Sake
  prefecture_id?: number,  // Brewery in this prefecture
  top_k?: number           // default 10, max 50
}
```

At least one of `axes`, `tags`, or `prefecture_id` must be present, or the tool returns an "empty filter" error.

**Output:** `Sake[]` matching all provided filters, each with the same shape as a `search_sakes_by_name` result. Order is implementation-defined for v0.1.0 (suggested: by Sakenowa `brand_id` ASC for determinism).

**SQL intent:** joined query against `sakes`, `flavor_profiles`, `breweries`, `sake_flavor_tags`. Axis filters become `WHERE flavor_profile.<axis> BETWEEN min AND max`. Tag filter requires all listed ids to be present (HAVING COUNT = N pattern, or repeated joins). Prefecture filter joins through brewery.

The `areaId: 0` sentinel is excluded.

### 6. `get_top_ranked`

**Purpose:** Return the top-N ranked Sakes for either the overall ranking or a specific prefecture's ranking, using the latest monthly snapshot the mirror has stored.

**Input:** `{ scope: 'overall' | 'prefecture', prefecture_id?: number, limit?: number /* default 10, max 100 */ }`. When `scope: 'prefecture'`, `prefecture_id` is required.

**Output:** `Array<Sake & { rank: number, year_month: string }>` ordered by `rank` ASC.

**SQL intent:** `rankings` table joined to `sakes` × `breweries` × `prefectures`. Filter by `scope`; latest `year_month` (the mirror stores only the latest, so this is essentially a straight read).

When `scope: 'prefecture'` with an invalid `prefecture_id`, return an empty list (not an error). When `scope: 'overall'` with `prefecture_id` supplied, ignore `prefecture_id` silently.

## Expected DB schema

The server expects the consumer's Postgres to have these **canonical Sakenowa-API-shaped** tables — the shape any Sakenowa mirror naturally takes. Columns the tools read are shown; consumer ingest-bookkeeping columns (`source`, `confidence`, `content_hash`, `updated_at`, `name_kanji`, …) may also be present and are ignored. `name_romaji` is a nullable consumer-enrichment column the tools read when present.

```
areas (
  area_id     int primary key,            -- Sakenowa area id; 0 = "その他" (Other) sentinel
  name        text not null               -- Japanese name, e.g. 北海道
)

breweries (
  brewery_id  int primary key,            -- Sakenowa brewery id
  name        text not null,
  name_romaji text,                        -- nullable enrichment
  area_id     int not null references areas(area_id)
)

brands (
  brand_id    int primary key,            -- Sakenowa brand id (a "sake")
  name        text not null,
  name_romaji text,                        -- nullable enrichment
  brewery_id  int not null references breweries(brewery_id)
)

flavor_charts (
  brand_id  int primary key references brands(brand_id),
  f1 numeric, f2 numeric, f3 numeric,      -- the six Sakenowa axes f1..f6 in [0,1]
  f4 numeric, f5 numeric, f6 numeric
)

flavor_tags (
  tag_id    int primary key,              -- Sakenowa tag id
  name      text not null
)
-- NOTE: there is no brand↔tag junction in the canonical mirror, so the
-- per-brand tag association is unavailable; see schema-audit-v0.1.1.md.

rankings (
  kind       text check (kind in ('overall', 'area')),
  area_id    int references areas(area_id),  -- NULL when kind = 'overall'
  rank       int not null,
  brand_id   int not null references brands(brand_id),
  score      numeric
)
```

This shape mirrors what the Yawaragi project's ingest produces; consumers with a differently-shaped mirror can either reshape their data or fork this server. v0.2 may introduce an env-configurable column-name map if multiple consumer ingests appear.

## Error handling

- Schema mismatch on startup: stderr log + non-zero exit. Better to fail loudly than serve garbage.
- "Not found" results (e.g. `get_sake_details(99999999)`): return an explicit `{ found: false, sake_id }` shape, not an error.
- Invalid arguments past Zod parse: return an MCP error response, not a thrown exception.
- DB connection failure mid-request: return an MCP error response with a clear message; do not crash the server.

## Versioning

- v0.1.0 ships the six tools above. Adding a tool, removing a tool, or changing a tool's shape is a minor-version bump.
- The DB schema expectations are part of the public contract. Changing them is a minor-version bump.
- Internal refactors with no observable change are patch bumps.

## Testing

- Unit tests for each tool's query function, with a stubbed DB client (`pg.Client`-shaped).
- Integration tests via testcontainers Postgres: seed a small fixture set covering each tool's interesting branches (no-match, ambiguous, multi-tag filter, areaId-0 exclusion). Tag with `.integration.test.ts`.
- A stdio-transport smoke test that boots the server, sends a `tools/list` JSON-RPC request, and verifies all six tools are advertised.

CI runs unit + integration on PRs.
