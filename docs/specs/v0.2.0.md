# v0.2.0 specification ‚Äî Streamable HTTP transport (additive)

This document specifies the v0.2.0 release. It is **additive**: the entire
v0.1.0 contract ‚Äî the six tools, their input/output shapes, the expected DB
schema, the read-only/stateless/domain-pure posture ‚Äî is unchanged and remains
authoritative in [`v0.1.0.md`](./v0.1.0.md). v0.2.0 adds a **second transport**
and nothing else.

## Goal

Let consumers that cannot keep a child process alive between requests (a
serverless web app, a long-lived network service) talk to the server over HTTP,
without disturbing the stdio path that Claude Desktop / IDE consumers rely on.

## Transport selection

The transport is chosen at startup from `MCP_TRANSPORT`:

| `MCP_TRANSPORT` | Meaning |
|---|---|
| unset / empty / `stdio` | **stdio** (default) ‚Äî bit-for-bit the v0.1.0 behaviour |
| `http` | **Streamable HTTP** |
| anything else | startup fails: explanatory message on stderr, non-zero exit |

`DATABASE_URL` is still required in both modes; its absence still fails loud on
stderr with a non-zero exit. `MCP_LOG_LEVEL` is unchanged and applies to both
modes (all logs to stderr).

### HTTP-mode configuration

Read only when `MCP_TRANSPORT=http`:

| Variable | Default | Purpose |
|---|---|---|
| `MCP_HTTP_PORT` | `3030` | listen port (integer 1‚Äì65535; invalid value fails loud) |
| `MCP_HTTP_HOST` | `0.0.0.0` | bind host |
| `MCP_HTTP_PATH` | `/mcp` | JSON-RPC endpoint path (a leading `/` is added if omitted) |

## HTTP transport contract

- Uses the MCP **Streamable HTTP** transport
  (`@modelcontextprotocol/sdk/server/streamableHttp` ‚Üí
  `StreamableHTTPServerTransport`). No new direct dependency ‚Äî the transport
  (and its `@hono/node-server` bridge) ships transitively with the SDK.
- **Stateless.** No `Mcp-Session-Id` sessions; each request is independent. A
  fresh MCP server + transport is created per request, all sharing the one
  Postgres pool created at startup. This suits a read-only wrapper with no
  per-connection state and avoids JSON-RPC id collisions between concurrent
  callers.
- **JSON responses.** `enableJsonResponse` is on: a JSON-RPC request gets a
  plain `application/json` JSON-RPC response. The Sakenowa tools are short
  synchronous reads, so SSE streaming is not used.
- Requests to a path other than `MCP_HTTP_PATH` get a JSON-RPC error envelope
  with HTTP 404.
- The same query functions and the `Db` seam serve both transports ‚Äî there is
  no transport-specific tool logic.

## Out of scope for v0.2.0

The OSS contract ‚Äî stateless, no auth, no business logic ‚Äî is unchanged. The
HTTP transport is a network surface but the project is still not an application
server.

- **Authentication / authorization.** The endpoint binds plain HTTP with no
  auth. Consumers put auth in front at deploy time (Cloudflare Tunnel, a reverse
  proxy with a bearer token, a platform gateway). The server cannot enforce it.
- **TLS.** The server binds plain HTTP; terminate TLS in front of it.
- **SSE / streaming responses** for long-running tool calls. The tools are short
  reads; synchronous responses are correct. Streaming is a later concern.
- **Per-request session state.** The Streamable HTTP spec optionally supports
  sessions; a stateless read-only wrapper does not need them.

## Versioning

Adding a transport is an additive, backward-compatible capability ‚Üí a
**minor** bump (0.1.0 ‚Üí 0.2.0). Existing stdio consumers need no config change.
The DB schema expectations and tool contracts are unchanged from v0.1.0.

- HTTP smoke (no Docker needed ‚Äî `tools/list` and a stubbed-`Db` tool call never

  `tools/call`, assert all six tools are advertised and the structured envelope
  comes back, and that an off-path `POST` returns a JSON-RPC 404.
- All v0.1.0 stdio tests stay green.
